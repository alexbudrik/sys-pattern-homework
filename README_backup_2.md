# Задание 1: Резервное копирование

Кейс:
Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования.

Необходимо описать, какие варианты резервного копирования подходят в случаях:

1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.
Для данного случая оптимально использовать полное резервное копирование (Full Backup), выполняемое один раз в сутки, например ночью после окончания рабочего дня.
При возникновении сбоя можно восстановить базу данных из последней полной резервной копии и получить её состояние на конец предыдущего дня. Такой способ отличается простотой восстановления, поскольку все данные находятся в одном архиве. Недостатком является большой объём хранимых данных и длительное время создания резервной копии.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.
В данном случае полного ежедневного бэкапа недостаточно, поскольку можно потерять данные за целый день.
Наиболее подходящим решением будет:
ежедневный полный бэкап;
инкрементные резервные копии каждый час;
либо использование журналов транзакций (WAL в PostgreSQL или бинарных логов в MySQL).
При восстановлении сначала разворачивается последняя полная копия, после чего применяются инкрементные изменения или журналы транзакций до нужного момента времени. Такой подход позволяет восстановить базу практически на любой момент времени и ограничить потерю данных одним часом или даже несколькими минутами.

1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных.
Да, такой сценарий возможен с использованием репликации базы данных.
Репликация предполагает наличие основного сервера (Master/Primary) и одного или нескольких резервных серверов (Replica/Slave), которые получают изменения практически в режиме реального времени. Если основной сервер выходит из строя, одна из реплик может быть быстро назначена новым основным сервером, а приложение автоматически переключится на неё.
Преимущества такого решения:
минимальное время простоя системы;
высокая доступность сервиса;
очень низкие потери данных (при синхронной репликации практически нулевые).
Однако важно понимать, что репликация не заменяет резервное копирование, поскольку ошибки или случайное удаление данных также будут скопированы на все реплики. Поэтому для финансовой компании рекомендуется использовать репликацию совместно с регулярными резервными копиями.

Вывод

Для финансовой компании рекомендуется следующая схема:
Ежедневный полный бэкап для восстановления данных за предыдущий день.
Инкрементные бэкапы каждый час или архивирование журналов транзакций для восстановления на конкретный момент времени.
Репликация Master–Slave для обеспечения высокой доступности и быстрого переключения при отказе основного сервера.
Регулярный мониторинг и проверка возможности восстановления резервных копий.

# Задание 2: PostgreSQL

2.1. С помощью официальной документации приведите пример команды резервирования данных и восстановления БД (pgdump/pgrestore).

В PostgreSQL для создания логической резервной копии можно использовать утилиту pg_dump.

Пример команды резервного копирования:

pg_dump -U postgres -h localhost -p 5432 -F c -b -v -f /backup/company_db.backup company_db

Где:

-U postgres — пользователь PostgreSQL;
-h localhost — сервер базы данных;
-p 5432 — порт;
-F c — формат custom, подходящий для восстановления через pg_restore;
-b — включить большие объекты;
-v — подробный вывод выполнения;
-f — путь к файлу резервной копии;
company_db — имя базы данных.

Для восстановления базы данных используется команда pg_restore.

Пример восстановления:

createdb -U postgres company_db_restore
pg_restore -U postgres -h localhost -p 5432 -d company_db_restore -v /backup/company_db.backup

Если необходимо восстановить базу с удалением старых объектов, можно использовать:

pg_restore -U postgres -h localhost -p 5432 -d company_db_restore --clean --if-exists -v /backup/company_db.backup

Таким образом, pg_dump используется для создания резервной копии, а pg_restore — для восстановления базы данных из архива.

2.1.* Возможно ли автоматизировать этот процесс? Если да, то как?

Да, процесс резервного копирования PostgreSQL можно автоматизировать. Для этого обычно создают скрипт, который запускает pg_dump, сохраняет файл с датой в имени и затем выполняется по расписанию через cron.

Пример скрипта:

#!/bin/bash

BACKUP_DIR="/backup/postgresql"
DB_NAME="company_db"
DB_USER="postgres"
DATE=$(date +%Y-%m-%d_%H-%M)

pg_dump -U $DB_USER -F c -b -v -f $BACKUP_DIR/${DB_NAME}_${DATE}.backup $DB_NAME

Пример задания в cron для ежедневного запуска в 02:00:

0 2 * * * /home/postgres/backup_company_db.sh

Также можно добавить удаление старых резервных копий, например старше 14 дней:

find /backup/postgresql -type f -name "*.backup" -mtime +14 -delete

Для надёжности автоматизированного процесса необходимо проверять наличие свежего бэкапа, размер файла, ошибки в логах и периодически выполнять тестовое восстановление в отдельной среде. Это важно, потому что сам факт создания файла ещё не гарантирует, что база действительно сможет быть восстановлена.

№ Задание 3: MySQL

3.1. С помощью официальной документации приведите пример команды инкрементного резервного копирования базы данных MySQL.

В MySQL инкрементное резервное копирование обычно выполняется с использованием бинарных журналов (Binary Logs). После создания полной резервной копии сохраняются только изменения, которые были записаны в бинарные логи.

Для создания полной резервной копии можно использовать:

mysqldump -u root -p --all-databases > full_backup.sql

После этого выполняется сохранение бинарных логов:

mysqladmin -u root -p flush-logs

Команда создаёт новый бинарный лог, а предыдущий файл содержит все изменения, произошедшие после последнего резервного копирования.

Для просмотра содержимого бинарного лога используется:

mysqlbinlog mysql-bin.000001 > incremental_backup.sql

Таким образом, файл incremental_backup.sql будет содержать все изменения, внесённые в базу данных после создания полной резервной копии.

Для восстановления необходимо:

Восстановить полный бэкап:
mysql -u root -p < full_backup.sql
Применить инкрементные изменения:
mysql -u root -p < incremental_backup.sql

Такой подход позволяет существенно уменьшить объём резервных копий и восстановить базу данных до состояния, максимально близкого к моменту сбоя.

3.1.* В каких случаях использование реплики будет давать преимущество по сравнению с обычным резервным копированием?

Репликация имеет преимущества в случаях, когда критически важно обеспечить непрерывную работу сервиса и минимизировать время простоя.

Основные преимущества реплики:

Высокая доступность системы.
При отказе основного сервера можно быстро переключить пользователей на реплику, не дожидаясь восстановления из резервной копии.
Минимальное время простоя.
Восстановление из резервной копии может занимать часы, особенно для крупных баз данных. Реплика позволяет возобновить работу практически сразу.
Минимальная потеря данных.
Реплика получает изменения практически в реальном времени, поэтому потеря данных ограничивается лишь задержкой репликации.
Распределение нагрузки.
Реплики могут обслуживать запросы на чтение (SELECT), снижая нагрузку на основной сервер и повышая производительность системы.
Возможность тестирования.
Реплика может использоваться для проверки обновлений, отчётности и аналитики без воздействия на рабочую базу данных.

Однако репликация не заменяет резервное копирование. Если данные были случайно удалены или повреждены на основном сервере, эти изменения будут автоматически переданы на реплику. Поэтому для надёжной защиты данных рекомендуется использовать одновременно и репликацию, и регулярные резервные копии.

Вывод

Для MySQL рекомендуется использовать комбинированный подход:

регулярные полные резервные копии (mysqldump);
инкрементное резервное копирование через бинарные журналы (mysqlbinlog);
репликацию Master–Slave для обеспечения высокой доступности и быстрого восстановления работы сервиса.

Такой подход позволяет минимизировать как потерю данных (RPO), так и время простоя системы (RTO).
