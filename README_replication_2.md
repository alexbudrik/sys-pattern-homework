# Задание 1: 

Преимущества схемы «активный master-сервер и пассивный репликационный slave-сервер»
В данной архитектуре все операции записи выполняются на master-сервере, а slave-сервер содержит его актуальную копию. Основным преимуществом такого подхода является повышение отказоустойчивости системы. Если основной сервер выходит из строя, реплика может быть быстро переведена в роль нового master, что позволяет минимизировать простой системы. Кроме того, наличие резервной копии данных снижает риск их потери. Репликация является одним из наиболее распространённых способов обеспечения надёжности баз данных.
Также slave-сервер можно использовать для выполнения резервного копирования, аналитических запросов и тестирования без дополнительной нагрузки на основной сервер.
Преимущества схемы «master-сервер и несколько slave-серверов»
Использование нескольких реплик позволяет не только обеспечить отказоустойчивость, но и значительно увеличить производительность системы. Запросы на чтение могут распределяться между несколькими slave-серверами, что снижает нагрузку на master и позволяет обслуживать большее количество пользователей одновременно.
Дополнительным преимуществом является высокая доступность данных. При отказе одной реплики остальные продолжают работать, сохраняя доступ к информации. Кроме того, наличие нескольких копий базы данных повышает надёжность хранения данных и предоставляет больше возможностей для резервного копирования и обслуживания системы без остановки работы.
Таким образом, схема master + slave в первую очередь обеспечивает резервирование и защиту от сбоев, а схема master + несколько slave дополнительно позволяет масштабировать чтение данных и эффективно распределять нагрузку между серверами, что особенно важно для высоконагруженных систем.

# Задание 1: 

Для базы данных, состоящей из таблиц «пользователи», «книги» и «магазины», можно применить комбинированный подход: вертикальный шардинг для разделения разных типов данных и горизонтальный шардинг для распределения больших таблиц по строкам.

Предположим, что база данных имеет следующие таблицы:

users:
user_id
name
email
password_hash
phone
address
created_at
books:
book_id
title
author
category_id
price
stock_qty
store_id
stores:
store_id
store_name
city
address
phone

Вертикальный шардинг

При вертикальном шардинге данные разделяются по смыслу и по частоте использования.

На отдельные базы можно вынести:

База users_main:
user_id
name
email
created_at

Эта база хранит основную информацию о пользователях, которая часто используется приложением.

База users_private:
user_id
password_hash
phone
address

Здесь хранятся чувствительные и менее часто используемые данные пользователей.

База books_catalog:
book_id
title
author
category_id
price

Эта база используется для отображения каталога книг.

База books_stock:
book_id
store_id
stock_qty

Эта база хранит информацию об остатках книг по магазинам.

База stores_db:
store_id
store_name
city
address
phone

Эта база хранит справочник магазинов.

Такое разделение снижает нагрузку на отдельные серверы. Например, при просмотре каталога пользователю не нужно обращаться к базе с остатками или к базе с приватными данными пользователей.

Горизонтальный шардинг

Горизонтальный шардинг целесообразно применить к таблицам users и books, так как они могут содержать большое количество строк.

Таблица users делится по диапазону user_id:
users_shard_1 хранит пользователей с user_id от 1 до 1 000 000;
users_shard_2 хранит пользователей с user_id от 1 000 001 до 2 000 000;
users_shard_3 хранит пользователей с user_id от 2 000 001 и выше.
Таблица books делится по category_id:
books_shard_1 хранит книги с category_id от 1 до 3;
books_shard_2 хранит книги с category_id от 4 до 6;
books_shard_3 хранит книги с category_id от 7 и выше.
Таблица stores обычно меньше по объёму, поэтому её можно не шардировать. Она может храниться на отдельном сервере как справочная таблица. При необходимости её можно реплицировать на другие серверы для ускорения чтения.

Принципы построения системы

В системе используется master-сервер, который принимает запросы от приложения и определяет, к какому шарду нужно обратиться. На master-сервере может находиться общее представление данных, например users_view или books_view, объединяющее данные из нескольких шардов.

Каждый shard-сервер хранит только свою часть данных. Для маршрутизации используются ключи шардирования:

для пользователей — user_id;
для книг — category_id;
для остатков книг — store_id и book_id.

Запросы на запись направляются в нужный шард в зависимости от ключа. Запросы на чтение также выполняются только на нужном шарде, что ускоряет работу системы, так как сервер обрабатывает меньший объём данных.

Блок-схема размещения данных

                         ПРИЛОЖЕНИЕ
                              |
                              v
                       MASTER-СЕРВЕР
          маршрутизация запросов и общее представление
                              |
        ------------------------------------------------
        |                      |                       |
        v                      v                       v
    USERS CLUSTER          BOOKS CLUSTER             STORES DB
        |                      |                       |
    -------------          -------------             -----------
    |     |     |          |     |     |             stores_db
    v     v     v          v     v     v
    U1    U2    U3         B1    B2    B3

U1 = users с user_id 1–1 000 000
U2 = users с user_id 1 000 001–2 000 000
U3 = users с user_id 2 000 001+

B1 = books с category_id 1–3
B2 = books с category_id 4–6
B3 = books с category_id 7+

stores_db = справочник магазинов

Дополнительно вертикальное разделение можно представить так:

USERS:
users_main      -> user_id, name, email, created_at
users_private   -> user_id, password_hash, phone, address

BOOKS:
books_catalog   -> book_id, title, author, category_id, price
books_stock     -> book_id, store_id, stock_qty

STORES:
stores_db       -> store_id, store_name, city, address, phone

Режим работы серверов

Master-сервер:
принимает запросы от приложения;
определяет нужный шард;
может хранить общие представления;
сам не хранит все данные полностью, а управляет доступом к шардам.
Shard-серверы:
работают в активном режиме;
каждый сервер отвечает за свою часть данных;
принимают операции чтения и записи только по своему диапазону данных.
Репликационные slave-серверы:
могут быть добавлены к каждому shard-серверу;
работают в пассивном режиме;
используются для резервного копирования, чтения и повышения отказоустойчивости.

Итог

В предложенной архитектуре вертикальный шардинг разделяет данные по смыслу и назначению, а горизонтальный шардинг распределяет большие таблицы по нескольким серверам. Это позволяет снизить нагрузку на отдельные серверы, ускорить выполнение запросов и повысить отказоустойчивость системы.
